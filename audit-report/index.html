Security Issue Distribution

Critical Findings (all resolved)

Critical Fixed Stripe Secret Key Fragment Logged to Console api/bookstore/checkout/route.ts:88-100
The first 14 characters of STRIPE_SECRET_KEY are logged on every checkout request: console.log('[shop/checkout] key prefix:', keyPrefix). This exposes the key environment (sk_live_ vs sk_test_) and partial key entropy to anyone with log access (Vercel logs, third-party monitoring).
Remove all secret key logging. Use a non-sensitive request ID (session UUID or user ID) for debugging traceability instead.
All debug console.log statements removed; checkout route rewritten with server-side Sanity canonical price validation. Issue #49 · commit 9e8de17.
Critical Fixed Mass Assignment Vulnerability — Job Application Route api/job-application/route.ts:15-70
Request body is parsed with await request.json() and fields are spread directly into a Sanity document with no Zod schema validation. An attacker can inject arbitrary fields (e.g., _type, approved: true) into the Sanity document store. The careers-application route has Zod validation — this route does not.
Mirror the /careers-application pattern: add a strict Zod schema validating name, email, position, URLs, and enumerating allowed fields only. Reject any unrecognized field.
Strict Zod schema added to job-application route; all fields explicitly typed and allowlisted; _type injection impossible. Issue #50 · commit 9e8de17.

High Severity Findings (all resolved)

High Fixed No Rate Limiting on 3 Public Submission Endpoints api/job-application · api/secure-contact · api/whistleblower
Whistleblower, secure-contact, and job-application routes accept unlimited POST submissions. The whistleblower endpoint is particularly sensitive — spam submissions could flood the Sanity dataset and overwhelm editorial staff review. Upstash is already integrated in the codebase for bookstore routes but not applied here.
Apply makeRatelimiter(5, 300) (5 requests/5 min per IP) from the existing bookstore rate-limit pattern to all three endpoints.
Upstash sliding-window rate limiters added to all three routes: checkSubmissionRate (5 req/300 s) for job-application and secure-contact; checkWhistleblowerRate (3 req/300 s, strictest) for whistleblower. Issue #60 · commit cb9d6d2.
High Fixed Timing Attack on Cron & Webhook Secret Comparison api/cron/cleanup-briefs/route.ts:6 · api/webhooks/supabase-order-update/route.ts:36 · api/bookstore/internal/send-email/route.ts
Both routes use plain string equality (!== `Bearer ${secret}`) to validate secrets. String comparison in JavaScript short-circuits on the first mismatched character, leaking timing information that can allow brute-force secret reconstruction via network timing attacks.
Replace with crypto.timingSafeEqual(Buffer.from(a), Buffer.from(b)) for all secret comparisons.
verifyBearerToken() using crypto.timingSafeEqual with fixed-length buffer padding added to cleanup-briefs, supabase-order-update, and internal/send-email. Issue #61 · commit cb9d6d2.
High Fixed Client-Controlled Stripe Price IDs in Checkout api/bookstore/checkout/route.ts:52, 65, 104
The stripePriceId is taken directly from the client-submitted cart body and passed to Stripe without server-side whitelist validation. A sophisticated attacker could substitute a price ID from a different product or a $0.01 price, purchasing books at arbitrary prices.
Maintain a server-side allowlist of valid price IDs. For each cart item, look up the price in Sanity/Stripe server-side to get the authoritative amount before creating the session.
Checkout route rewritten: fetchCanonicalPricing() batches a Sanity GROQ query (cache:'no-store') for all cart book IDs; resolveCanonicalPriceId() returns the authoritative stripePriceId from Sanity — client-supplied IDs are completely ignored. Issue #62 · commit cb9d6d2.
High Fixed Insufficient Authorization on Whistleblower Endpoint api/whistleblower/route.ts:5-54
The endpoint accepts arbitrary submissionId, organization, and severity values from anonymous users with no validation, no CAPTCHA, and no rate limiting. Severity escalation allows attackers to flood the high-severity queue.
Add Zod validation (enumerate severity enum values), rate limiting per IP, and consider a privacy-preserving proof-of-work or Turnstile CAPTCHA.
Full Zod schema with enumerated severity and category; submissionId and submittedAt are server-generated (randomUUID(), new Date().toISOString()); all fields length-bounded; rate limiting applied. Issue #63 · commit cb9d6d2.
High Fixed TOCTOU Race Condition on Download Counter api/bookstore/download/route.ts:89-98
The download counter check-then-increment is not atomic. Between the check (download_count < max_downloads) and the increment, a race condition allows a user to trigger multiple concurrent downloads, exceeding the purchased limit.
Use a Supabase stored procedure with a FOR UPDATE lock, or increment before generating the signed URL and roll back on error.
PL/pgSQL stored procedure increment_download_if_allowed() with FOR UPDATE row lock replaces check+increment; download route calls .rpc('increment_download_if_allowed') — signed URL only generated if procedure returns true. Migration 20260522000001. Issue #64 · commit cb9d6d2.
High Fixed Service Role Client Bypasses RLS — Undocumented api/bookstore/my-downloads/route.ts:14-38
The route uses shopServiceClient (Supabase service role) which bypasses all Row Level Security policies. If Clerk auth validation is ever misconfigured or bypassable, all customers' download records are exposed. No RLS policy documentation exists in supabase/migrations/.
Document RLS policies in migration files. Use the anon/authenticated client wherever possible. Service role should only be used for server-side admin operations with explicit justification.
Migration 20260522000002_rls_service_role_documentation.sql added: documents per-route service-role justifications, confirms RLS enabled on all user-facing tables, and details the Clerk+Supabase JWT upgrade path. Code comments added inline. Issue #65 · commit cb9d6d2.

Additional Findings — Post-PR Audit (all resolved)

Four additional vulnerabilities were discovered during code review of the security-issues PR and resolved before merge.

High PR Audit Fixed Algolia Sync Webhook Fails Open When Secret Not Configured api/algolia-sync/route.ts
When SANITY_WEBHOOK_SECRET is not set in the environment, the original code skipped HMAC validation entirely and processed the webhook payload. Any caller could poison or delete Algolia index records without authentication.
Fail closed: reject the request with 401 immediately if the secret is not configured.
Fail-closed guard added: missing secret → immediate 401 with console.error log. Signature validation always runs when secret is present. Commit cb9d6d2.
High PR Audit Fixed Author Role Authorization Bypass — Any Author Could Modify Any Order api/portal/orders/[id]/status/route.ts
The author ownership check fetched order_items and selected sanity_book_id but only tested items.length > 0. Since every real order has at least one item, any author-role user could PATCH any order in the system (cancel, mark shipped, issue refund, etc.).
Run a GROQ query that resolves the Clerk user's Sanity author ID and counts how many of the order's book IDs reference that author. Return 403 if count is zero.
authorOwnsOrderItem() added: single GROQ query cross-references Clerk user → Sanity author → book IDs in order items; cache:'no-store' ensures freshness. Commit cb9d6d2.
High PR Audit Fixed Unvalidated Tracking URL Injected into Customer Emails api/portal/orders/[id]/status/route.ts
tracking_url was validated only as z.string().optional(). The value is stored in orders.shipping_tracking_url and injected as a raw href into customer shipment emails — a portal user who can set status:'shipped' could write an arbitrary phishing URL into branded customer email.
Validate tracking_url against an HTTPS-only carrier domain allowlist before accepting it.
ALLOWED_TRACKING_HOSTS set of 18 known carrier/aggregator hostnames added; Zod .refine(isAllowedTrackingUrl) rejects any URL not on the allowlist or not using HTTPS. Commit cb9d6d2.
High PR Audit Fixed Editor Role Bypasses All Order Authorization Guards api/portal/orders/[id]/status/route.ts:119
The route is documented as accessible to admin, sales, and author only. The editor role passed role !== 'admin', was not caught by isSalesOnly(), and was not caught by role === 'author' — giving editors unrestricted ability to update or refund any order in the system.
Add an explicit role allowlist immediately after role resolution so unknown roles receive 403 before reaching business logic.
Explicit allowlist guard added: if (role !== 'admin' && role !== 'sales' && role !== 'author') return 403 — any role not in the set is denied before any business logic runs. Commit 0b42afb.

SEO/AEO Audit — Security Finding (resolved)

One high-severity security vulnerability was discovered during the SEO/AEO audit pass (feature/seo-aeo-audit-68-79) and fixed before merge.

High SEO Audit Fixed XSS via JSON-LD Slug Injection in ClaimReview Schema lib/factCheck/claimReviewJsonLd.ts:20
The buildClaimReviewJsonLd() function constructs pageUrl by directly interpolating fc.slug.current: `${SITE_URL}/fact-check/${fc.slug.current}`. This URL is embedded in JSON via JSON.stringify(), which does not escape HTML angle brackets. The JSON-LD is then rendered via dangerouslySetInnerHTML in a <script> block. A Sanity fact-check document with a slug containing </script><script>alert(1)</script> would terminate the script block and inject arbitrary JavaScript — a stored XSS exploitable by any Sanity editor with document creation rights.
Sanitize the slug before URL construction: strip HTML-special characters (<, >, ", ', &) from the slug. Valid Sanity slugs are [a-z0-9-] only, so stripping is safe and non-destructive for well-formed content.
Slug sanitized in buildClaimReviewJsonLd() via .replace(/[<>"'&]/g, '') before URL construction — any HTML-special characters are stripped. Confidence: 8/10. Commit on feature/seo-aeo-audit-68-79.

Medium Severity Findings

Medium Missing Security Headers (CSP, X-Frame-Options, HSTS) next.config.ts (missing headers() function)
No Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, or Referrer-Policy headers are configured. Without CSP, any XSS vulnerability has maximum impact. Without X-Frame-Options, the site can be embedded in iframes for clickjacking.
Add a headers() function to next.config.ts with Content-Security-Policy, X-Frame-Options: SAMEORIGIN, X-Content-Type-Options: nosniff, and Referrer-Policy: strict-origin-when-cross-origin.
Medium File Upload MIME-Only Validation (No Magic Bytes) api/portal/upload-image/route.ts:34-43
File uploads validate MIME type from the Content-Type header only. An attacker can rename a malicious file as .jpg and bypass the check, since MIME type comes from the request, not the file content.
Add file-type npm package to read the first 12 bytes and validate actual file magic bytes. Reject files whose magic bytes don't match allowed image types.
Medium Guest Download Token Doesn't Validate Original Purchaser Email api/bookstore/download/guest-resend/route.ts:73-119
An attacker who knows a valid order number can trigger a download link resend to any email address, potentially redirecting download access. The endpoint validates the order exists but doesn't confirm the requested email matches the original purchaser.
Store the original guest email in the order record and compare it against the email in the resend request before generating a new token.
Medium Missing Stripe Idempotency Keys on Checkout Session api/bookstore/checkout/route.ts:149
Stripe checkout sessions are created without idempotency keys. Network failures that trigger retries create duplicate sessions, potentially allowing a user to complete two payments for the same cart.
Pass { idempotencyKey: \`checkout-\${userId}-\${cartHash}\` } as the second argument to stripe.checkout.sessions.create().
Medium Permissive Email Validation Regex api/bookstore/checkout/route.ts:40 · api/bookstore/newsletter/route.ts:22
The regex /^[^\s@]+@[^\s@]+\.[^\s@]+$/ allows technically invalid emails like test@..com. Inconsistent with the Zod .email() validation used elsewhere in the codebase.
Replace all custom email regexes with z.string().email() from Zod for consistent, RFC 5322-compliant validation.
Medium Missing CAPTCHA on Sensitive Public Endpoints api/careers-application · api/whistleblower · api/secure-contact
High-sensitivity endpoints (whistleblower submissions, secure contact, careers) accept anonymous form submissions with no bot protection. Automated abuse can flood queues and waste editorial resources.
Integrate Cloudflare Turnstile (free, privacy-preserving) or hCaptcha with server-side token verification on all three endpoints.
Low Stripe Error Details Returned in 500 Response Body api/bookstore/checkout/route.ts:176-181
err.message is returned directly in the 500 JSON response, which may include Stripe internal error details that reveal payment processing internals.
Return a generic user-facing message for 500 errors, log the full error internally (or to Sentry). Only differentiate error types (validation vs payment) without exposing internal details.

Critical Theme Toggle Button Has No aria-label components/global/ThemeToggle.tsx:12-20
The dark/light mode toggle button renders a Sun or Moon icon with no accessible text. Screen readers announce it as "button" with no purpose — users cannot toggle the theme.
Add aria-label="Toggle theme" (or dynamic: aria-label={`Switch to ${theme === 'dark' ? 'light' : 'dark'} mode`}) to the button element.

Critical Form Labels Not Associated with Inputs — ApplicationForm (WCAG 1.3.1) components/careers/ApplicationForm.tsx:79-80
Form labels exist as <label> elements but lack htmlFor attributes. Inputs lack matching id attributes. Screen readers cannot associate labels with their controls — users don't know what each field is for.
Add htmlFor="fullName" to the label and id="fullName" to the input for every form field pair throughout the form.
Critical Error Messages Not Associated with Form Fields (WCAG 3.3.1) components/careers/ApplicationForm.tsx:76-179
Validation error messages are rendered as sibling paragraphs but not linked to their fields via aria-describedby. Screen reader users don't know which field has an error or what the error is.
Add aria-describedby="fullName-error" to each input and id="fullName-error" to the corresponding error paragraph. Add aria-invalid={!!errors.fullName} to invalid fields.
Critical Portal Modal Icon Buttons Without Labels components/portal/PitchQuickViewModal.tsx:145-147, 202-208, 227-233
Three icon-only buttons in the pitch modal (Add reference, Remove reference, Close) have no aria-label. Portal users who rely on screen readers cannot operate any modal actions.
Add descriptive aria-label to each: "Close pitch details", "Add reference link", "Remove this reference link".

High: Navigation & Modal Accessibility

High No Skip Navigation Link (WCAG 2.4.1) app/layout.tsx:115-119
There is no skip-to-main-content link anywhere in the layout. Keyboard-only and screen reader users must navigate through the entire navigation bar on every page load — potentially 20+ tab stops before reaching content. This is a WCAG Level A failure.
Add <a href="#main-content" className="sr-only focus:not-sr-only">Skip to main content</a> as the first element in <body>, and add id="main-content" to the <main> element.
High Modal Lacks Focus Trap (WCAG 2.4.3) components/portal/PitchQuickViewModal.tsx:107-276
When the pitch modal opens, focus is not trapped within it. Keyboard users can tab into background content behind the modal overlay, causing confusion about context and potentially activating hidden interactive elements.
Use the Radix UI Dialog primitive (which already handles focus trapping correctly) or implement a focus-trap package. Restore focus to the trigger element on modal close.
High Navigation Not Structured as Semantic List (WCAG 1.3.1) components/global/Nav.tsx:49-74
Primary navigation categories are rendered as <div> containers with ClientSideRoute links, not as a <ul>/<li> list. Screen readers don't announce the number of items or provide list navigation shortcuts.
Wrap category links in <ul role="list"> with <li> wrappers. Add aria-label="Article categories" to the <nav> element.
High Table Headers Missing scope Attribute (WCAG 1.3.1) components/admin/ApplicationsTable.tsx:83-354
The applications data table uses <th> header elements without scope="col" attributes. Screen readers cannot associate header cells with data cells, making the table difficult or impossible to interpret.
Add scope="col" to all column header <th> elements. For row headers (if first cell identifies the row), add scope="row".

Multiple API routes contain console.log() statements that run in production. The checkout route specifically logs Stripe key prefixes, cart item details, and user identifiers — all visible in Vercel function logs to anyone with log access. This is both a security risk and performance overhead.

Critical Widespread as any Type Assertions Bypass TypeScript Safety (news)/articles/[slug]/page.tsx:108, 115, 119, 155, 271, 294, 302 · author/[slug]/page.tsx:156, 165 · (news)/page.tsx:173, 180
Over 15 instances of as any type assertions across news pages and article pages, primarily around Sanity image objects and author data. This completely eliminates TypeScript's safety guarantees and has likely already masked runtime bugs where undefined properties are accessed.
Generate proper types from the Sanity schema using the existing TypeGen setup. Create a strongly-typed SanityImageObject type and update urlForImage() to accept it. Enable "strictNullChecks": true in tsconfig.json.
Critical Cart State vs Server Truth Mismatch at Checkout (user)/bookstore/cart/page.tsx:48-70 · lib/bookstore/cart.ts
Cart state exists in three out-of-sync places: Zustand (localStorage), Stripe checkout session, and Supabase orders table. Prices in localStorage can go stale if Sanity book prices change. A user who adds books, leaves, returns after a price change, and checks out will be charged the cached price, not the current price.
In the checkout API route, re-fetch book prices from Sanity/Stripe server-side for each cart item before creating the session. Never trust client-submitted prices.
Critical Missing Revalidation Strategy — ISR Cache Never Expires (user)/bookstore/page.tsx:211-221 · lib/sanity/lib/fetch.ts
Sanity fetches use tags ('book', 'bookGenre') but no route calls revalidateTag() or revalidatePath() after mutations. Book data can remain indefinitely stale in the ISR cache after content editors publish changes in Sanity Studio.
Configure the existing /api/revalidate webhook to call revalidateTag('book') on Sanity publish events. Set revalidate: 3600 as a fallback ceiling on all book queries.

High: Technical Debt

High Zustand Cart Store Missing Size Limits & Persistence Debouncing lib/bookstore/cart.ts
The cart Zustand store has no maximum item count limit and writes to localStorage on every state change with no debouncing. With large carts, this causes excessive synchronous localStorage writes. No cleanup or unsubscribe pattern means potential memory leaks in long-lived sessions.
Add MAX_CART_SIZE = 50, debounce persistence writes by 300ms, and ensure useEffect cleanup patterns in components that subscribe to the store.
High Incomplete Supabase RLS Policies Not Documented supabase/ (missing migration RLS policies) · api/bookstore/my-downloads/route.ts
No RLS policy definitions exist in the supabase/ directory. The service role client is used across download routes, which bypasses all RLS. Without documented RLS policies, there is no way to audit or verify data access controls are correctly configured in the Supabase dashboard.
Create RLS policy migration files in supabase/migrations/. Document all intended access patterns (users see own orders, admins see all). Use the authenticated client wherever RLS should apply.
High useCache Enabled in Config But Not Used Anywhere next.config.ts:53 · lib/sanity/lib/fetch.ts
The experimental useCache: true feature is enabled, enabling the 'use cache' directive and cacheTag()/cacheLife() APIs from next/cache. However, no route or function in the codebase actually uses this directive. The feature has non-zero overhead for being enabled with no benefit.
Either annotate expensive Sanity fetch wrappers with 'use cache' and cacheTag() to gain the intended benefit, or remove useCache: true from config until ready to implement.
High TypeScript strictNullChecks Disabled tsconfig.json:17
strictNullChecks is commented out in tsconfig.json. This means null and undefined are assignable to any type, making the TypeScript types lie about nullability throughout the codebase. This is likely the root cause of the widespread as any patterns.
Enable "strictNullChecks": true and fix the resulting type errors. Start with a partial enablement using // @ts-strict-ignore on files needing time, and incrementally clean up.

Medium BookBuySection Forces Entire Buy Flow to Client-Side components/bookstore/BookBuySection.tsx:1
BookBuySection is marked 'use client' and used inside a server component page. This forces the entire price display, format selection, and buy flow to hydrate client-side. Static price/format information that could be server-rendered is delayed until hydration completes.
Split the component: render static pricing data as a server component, keep only interactive elements (quantity inputs, NYOP field, cart button) as client components using composition patterns.

The logic for determining whether to use stripeProductId vs stripePriceId for name-your-own-price items is duplicated and inconsistently implemented across the client component and the API route. This is a maintenance hazard and a source of potential price bypass bugs.
Create a shared getStripeIdForFormat(format, isNyop) utility function with unit tests covering all cases. Single source of truth for this logic.
Medium N+1 Risk in Deep GROQ Reference Chains lib/sanity/lib/queries.ts (multiple queries)
Several GROQ queries chain deep reference dereferences: keyEvent[]->, relatedArticles[]->, categories[]->. While GROQ batches these internally, unguarded array expansions (e.g., *[_type == 'article' && author._ref == ^._id]) on author pages can return arbitrarily large result sets.
Limit array derefs: relatedArticles[0..3]->. Add [0..49] bounds to reverse-lookup queries on author pages. Profile with Sanity's API metrics dashboard.
Medium No Error Monitoring / Structured Logging Infrastructure All API routes and page components
The codebase has no integration with Sentry, Datadog, or any structured logging service. Production errors are only visible in Vercel function logs (which don't aggregate, alert, or retain history well). Checkout failures, download errors, and Stripe errors go undetected until a user complains.
Integrate Sentry for Next.js (@sentry/nextjs). Configure error boundaries to Sentry.captureException(). Add Vercel log drains to a logging service for structured queries on production issues.
Medium Missing Error Boundaries on Key Pages (user)/bookstore/book/[slug]/page.tsx:199-414
The book detail page makes parallel Sanity fetches but has no error boundaries. If the reviews fetch fails (network timeout, Sanity quota), the entire page 500s instead of gracefully showing the book with a "reviews unavailable" state.
Wrap the reviews section in a <Suspense> boundary with an error.tsx sibling, or use React's <ErrorBoundary> component to catch and display fallback UI independently from the main content.
Low Bookstore Conversion Events Not Tracked in Analytics components/bookstore/AddToCartButton.tsx · api/bookstore/checkout/route.ts
Cart adds, checkout initiations, purchase completions, and download events are not tracked in the analytics pipeline. Without conversion funnel data, it's impossible to identify where users drop off or optimize the purchase experience.
Add gtag('event', 'add_to_cart', {...}) events to cart actions, and a purchase event (with revenue and items) in the order-success page.
